chaos-jungle

A Generic Chaos Engineering Framework

Inject network delays  ·  storage corruption  ·  packet loss
on any Linux machine — locally, via SSH, or via HTTP daemon


Python 3.9+ Linux target No agent required Open source

What is Chaos Engineering?

The practice of deliberately injecting failures into a system
to discover weaknesses before they cause outages in production.


Without chaos testing
Failures are discovered in production, under load, at the worst time.
With chaos testing
Weaknesses are found in a controlled experiment, with full observability.

Hypothesis
→
Inject Fault
→
Measure Impact
→
Fix & Learn

Why chaos-jungle?

  • No agent on the target — uses existing tools (tc, dd)
  • Any target — local, SSH, HTTP daemon, containers
  • Automatic revert — always cleans up, even on crash
  • Built-in metric collection — baseline vs chaos comparison
  • Persistent SQLite session DB — every run is recorded
  • Portable CSV / JSON export — share results anywhere
  • Live dashboard — no extra dependencies
Origin
Based on the RENCI/PEARC '19 experiment framework.
Generalised into a reusable Python package that works
with any workflow, not just Pegasus WMS.

Fault Types

FaultWhat it doesRequired paramsTool used
NetworkDelay Add latency to all outgoing packets delay e.g. "100ms" tc netem
NetworkLoss Drop a % of packets rate e.g. "5%" tc netem
NetworkCorrupt Corrupt a % of packets (breaks checksum) rate e.g. "1%" tc netem
NetworkDuplicate Duplicate a % of packets rate e.g. "0.5%" tc netem
StorageCorrupt Flip bits in files on disk (via crontab) pattern, directory dd + cj_storage
SilentNetworkCorrupt Corrupt packets silently — valid checksum rate (1-in-N) BPF / XDP

All faults auto-detect the network interface via ip route if iface= is not given.
All faults are always reverted in a finally block — even on exception.

Targets — Where to inject

LocalTarget
Run on this machine. No connection needed.
Good for development and single-node tests. 

target = LocalTarget()
SSHTarget
Run on a remote machine over SSH.
Auth: key file, ssh-agent, or password. 

target = SSHTarget("worker1", user="ubuntu")
# or explicit key
target = SSHTarget("worker1", user="ubuntu",
                   key="~/.ssh/id_ed25519")
HTTPTarget (Daemon)
Control a remote machine via REST API.
Use when SSH is blocked or multiple controllers needed. 

# on the remote machine:
# cj-daemon --port 7777 --token secret

target = HTTPTarget("http://worker1:7777",
                    token="secret")

@chaos_measure — Auto-record results

Wraps a function with chaos and automatically saves its return dict as session metrics. 


from chaos_jungle.decorators import chaos_measure
from chaos_jungle.faults import NetworkDelay

@chaos_measure(
    NetworkDelay("100ms", jitter="10ms"),
    scenario_name="E1",
    conflict="warn",
)
def run_experiment():
    result = run_pipeline()
    return {                          # dict → auto-saved to DB
        "files_transferred": 120,
        "retries":             3,
        "throughput_mbps":   42.1,
    }

summary = run_experiment()
print(summary["duration_s"])          # how long chaos was active
print(summary["fn_result"])           # the dict above

Custom metric


from chaos_jungle.metrics import Metric

class RetransmitRate(Metric):
    name = "tcp_retrans"

    def collect(self, target) -> dict:
        _, out, _ = target.run(
          "awk '/^Tcp:/{getline; "
          "print $12}' /proc/net/snmp"
        )
        return {"retransmits": int(out.strip())}

SSH Authentication

Auth is attempted in order — same as the OpenSSH client:

  • 1
    Explicit key file — SSHTarget("h", user="u", key="~/.ssh/id_ed25519")
  • 2
    SSH agent — if ssh-add was used, picked up automatically
  • 3
    Default key search — Paramiko tries ~/.ssh/id_rsa, id_ecdsa, id_ed25519
  • 4
    Password — SSHTarget("h", user="u", password="...")


# most common — agent / default key
SSHTarget("worker1", user="ubuntu")

# password-only (no key)
SSHTarget("worker1", user="ubuntu",
          password="secret",
          allow_agent=False,
          look_for_keys=False)
Sudoers setup on target:
echo "ubuntu ALL=(ALL) NOPASSWD:
/sbin/tc, /bin/dd, /usr/sbin/filefrag" \
| sudo tee /etc/sudoers.d/chaos-jungle

When to use the Daemon

Use SSHTarget when:
  • SSH keys are already set up
  • Port 22 is open
  • One-off or short experiments
  • HPC / cloud VMs

Use Daemon (HTTPTarget) when:
  • Port 22 is blocked by firewall
  • Multiple controllers share one target
  • Container / Kubernetes pod
  • REST API integration (Ansible, CI/CD)
  • No SSH key management needed

# on the target machine — once
cj-daemon --port 7777 --token mysecret

# from anywhere
chaos-jungle start \
  --scenario net-delay \
  --delay 100ms \
  --target http://worker1:7777

)

Guardrails & Input Validation

Guardrails

Checked before any fault starts:

  • Two tc-netem faults on same interface → ConflictError
  • tc rule already active from previous run → ConflictError
  • StorageCorrupt + NetworkCorrupt together → warn


runner = ChaosRunner(
    scenario, target,
    conflict="warn",   # or "raise" / "force"
)

Input validation

Caught at __init__ time — not at runtime:


# ValueError: 'delay' must be like
# '100ms', '1s' — got 'hundred'
NetworkDelay("hundred")

# ValueError: 'rate' must be like
# '5%', '0.5%' — got '5'
NetworkLoss("5")

# ValueError: 'directory' must start with '/'
StorageCorrupt("*.pdb", "data/")

# ValueError: SSHTarget 'key' not found
SSHTarget("h", user="u",
          key="~/.ssh/missing")

ExperimentSuite — Multi-node Parallel Chaos


# suite.yml
duration: 10m
conflict: raise
auto_install: true

experiments:
  - name: control
    target: local
    faults: []

  - name: net-delay
    target: ssh://ubuntu@node1
    faults:
      - kind: NetworkDelay
        delay: 100ms
        jitter: 10ms

  - name: packet-loss
    target: ssh://ubuntu@node2
    faults:
      - kind: NetworkLoss
        rate: 5%

  - name: storage-corrupt
    target: ssh://ubuntu@node3
    duration: 5m
    faults:
      - kind: StorageCorrupt
        pattern: "*.pdb"
        directory: /scratch/data
      

chaos-jungle suite --config suite.yml
      

suite = ExperimentSuite(duration="10m")
suite.add(Scenario("control", []),
          LocalTarget())
suite.add(Scenario("delay",
          [NetworkDelay("100ms")]),
          SSHTarget("node1", user="ubuntu"))

results = suite.run(parallel=True)
ExperimentSuite.print_summary(results)
All experiments start simultaneously.
Each runs independently on its own target.
Results stored in the same DB.

System State During Experiment

Configurations applied to the target that persist if not reverted:

FaultState createdReverted by
NetworkDelay/Loss/Corrupttc qdisc rule on interfacerunner.stop()
StorageCorruptcrontab entry + deployed scriptsrunner.stop()
SilentNetworkCorruptBPF/XDP hook attachedrunner.stop()
All faultsSession record in chaos_jungle.dbalways

If experiment crashes:
tc rules stay active. Crontab keeps running.
Session stuck as running in DB.
Recovery:
chaos-jungle stop — reverts most recent
sudo tc qdisc del dev eth0 root
crontab -l | grep -v cj_storage | crontab -

Full Experiment Workflow

1. Define   Scenario + Faults + Target + Metrics
↓
2. Preflight   check deps · guardrails · interface detection
↓
3. Inject   baseline metrics → chaos ON → run workload → chaos metrics
↓
4. Revert   tc qdisc del · crontab remove · record results
↓
5. Analyse   dashboard · export CSV / JSON · chaos-jungle fetch
